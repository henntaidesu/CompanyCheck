# 日本企业求职背景调查报告（Company Background Dossier）

把「某家日本公司值不值得去应聘」这一问题，做成一份**结构统一、量化可比、视觉精美**的单文件 HTML 报告。范本见 `assets/template.html`（渡辺電機工業株式会社，已填充的样板），量化评分口径见 `assets/scoring-schema.json`。

## 何时使用
- 用户给出一家公司（名字 / 官网 / マイナビ・リクナビ链接）并想做求职背调。
- 用户想处理 `list.txt` 里排队的公司（一行一家：`公司名 官网`）。
- 用户问「这家公司稳不稳 / 待遇如何 / 好不好进 / 该不该去」。

## 输入
1. **公司名**（日文正式名，如「渡辺電機工業株式会社」）。
2. **官网 URL**（若只给名字，先搜官网）。
3. **目标岗位**（可选；如「設計開発エンジニア」。没有就按新卒/主力技术岗写）。

## 产出（两个文件，都放在本项目根目录）
1. `<公司名>.html` —— 自包含报告（内联 CSS/JS，无外链，直接双击可看）。
2. 更新 `经营财务稳健度评分.json` —— 在 `companies[]` 追加/更新这家公司的评分条目（供横向对比与复用）。

## 工作流程

### 1 · 调研（用 WebSearch / WebFetch）
按下列来源类型逐一取证，**记录每个数字的出处**（报告 §9 要列全）：

| 类型 | 典型站点 | 取什么 |
|---|---|---|
| 官网 | `*.co.jp` | 会社概要、沿革、代表者、従業員数、拠点、取引銀行、产品线、认证 |
| 官报决算 | catr.jp / バフェット・コード | 当期纯利益、利益剩余金、总资产（多期，用于 CAGR 与亏损年数） |
| 采用情报 | 公司 recruit 站 / マイナビ / リクナビ | 初任给、赏与、昇给、手当、年间休日、募集要项、选考流程 |
| 口碑 | OpenWork / 就活会议 / 転職会議 / ワンキャリア | 年收、加班、有薪假消化率、离职率、员工评价、求める人物像 |
| 年收 | doda / 各エージェント | 平均年收、年龄层区间 |

调研纪律：
- **样本量小要显式标注**（如「OpenWork 仅3人」）——报告里反复提醒「仅供参考、勿当定论」。
- **警惕同名公司混淆**：把不属于该公司的业务/描述剔除，以官网+官报为准（见范本 §3 的 caution）。
- **代表者/职位信息以官网为准**，求职平台常滞后。
- 缺失数据就标「数据不足 / 未公开」，不要编。

### 2 · 财务评分（`scoring-schema.json` 的 metrics）
用 8 项指标（自有资本比率、留存缓冲倍数、ROA、纯利益率、总资产 CAGR、留存 CAGR、盈利连续性、抗风险恢复力）各打 0–5 分，按权重加权归一到 0–100，对照 `rating_bands` 得 grade（A+…E）。缺失指标置 null 并对剩余权重重新归一。营业额是近似值时把 confidence 标 low。

### 3 · 求职者综合评分（`scoring-schema.json` 的 overall_model）
5 个维度按权重加权到 0–100：财务稳健度 30 / 待遇 22 / 成长 20 / 工作环境 18 / 进入门槛 10。财务维度直接取第 2 步的 weighted_total；其余维度据 §4/§6/§7 的信息打分，样本小的维度 confidence 标低。对照 `grade_bands` 得综合 grade。**「进入门槛」衡量好不好进，与公司质量独立** —— 别混为一谈。

### 4 · 渲染 HTML（照抄 `assets/template.html` 的结构与 CSS）
`assets/template.html` 是可直接复用的设计系统，**整段 `<style>` 和 `<script>` 原样保留**，只替换内容。十个章节固定：

- **Nameplate 头部**：eyebrow、公司名 h1、假名+英文 reading、官网链接、`nameplate-facts`（创业/资本金/代表/员工数/总部/认证）、`score-badge`（综合分+5维度分，可点开 modal 看依据）。
- **§1 公司做什么** — lede + `card-grid`（业务板块）+ `tl` 沿革时间线 + 一句话 `note`。
- **§2 经营与财务** — `scorecard`（大分+进度条）+ 8 指标量化表 + `stat-row` 原始数据 + 多期决算表 + 解读 `note`。
- **§3 产品与市场** — `split`（HW/SW 或 元器件/系统两栏）+ 市场定位 note + 同名混淆 caution。
- **§4 新卒待遇** — 初任给表、赏与/昇给/休日 stat-row、手当表、福利、整体年收参考。
- **§5 岗位详情** — `split` 硬件/软件方向拆解。
- **§6 选考流程** — `flow` 步骤条 + stat-row + `pc`（契合/不契合两栏）+ 难度 note。
- **§7 真实工作环境** — stat-row + `pc`（正面/负面）+ 「如何看待」caution。**必须平衡视角、标注样本量小。**
- **§8 求职者行动建议** — 志望动机切入点 + `ask` 面试逆问题清单 + 入职前核实事项 + 综合结论 note。
- **§9 信息来源** — `src-list` 列全部来源链接+域名。
- **§10 待确认事项** — 若干 `caution`，列出估算口径、滞后信息、待核实点。

替换要点（把范本里这些值换成新公司的）：
- `<title>`、`.eyebrow`/`h1`/`.reading`/`.site-link`
- `.nameplate-facts` 六个 fact；`.score-badge` 的分数、grade、5个 `sb-break` 维度分
- 底部 `<script>` 里的 `DIMS` 对象（5个维度的标题/分数/权重/理由/来源）——务必与 score-badge 对齐
- 各章节正文、表格、来源
- 页脚公司名

保持不变：整套 CSS 变量与深浅色主题、组件 class、modal 交互 JS 逻辑（只改 DIMS 数据）。

### 5 · 更新评分 JSON
把这家公司作为一个新条目加进 `经营财务稳健度评分.json` 的 `companies[]`（复制范本条目为模板，填 `raw` / `metrics` / `weighted_total` / `grade` / `overall`）。这样多家公司可横向对比。

## 质量基准
- 每个关键数字都能追溯到 §9 的某个来源；估算值明确标注（如营业额近似、纯利益率 confidence low）。
- 口碑类结论一律附样本量与「因部门/上司而异」的免责说明——不据小样本下定论。
- 视觉与范本一致：深浅色都要正常、移动端可读、表格能横向滚动。
- 报告是**给求职者做决策用的平衡视角**，优点和劝退点都要摊开，不做软文。
- 页脚保留免责声明：基于公开信息整理、仅供参考、关键决策前与官方核实。
